# 官网内容管理系统（CMS）— 思维对齐文档

> 生成日期: 2026-06-23
> 对齐状态: ✅ 已确认

---

## 原始需求

每次更新官网内容（新增项目、修改文案等）都需要经过"改代码 → git 提交 → Vercel 部署"的流程，太重。希望搭建一个后台管理系统，管理员在网页端直接编辑内容，发布后刷新即可看到更新，无需手动改代码。

---

## 理解确认

- **核心需求**：为 Aetherwit 官网搭建内容管理系统（CMS），管理员在网页后台直接编辑全站内容，发布后刷新即可生效，彻底告别"改代码 → git → 部署"的手动流程
- **目标用户**：用户本人和合伙人（2-3 人的管理员）
- **成功标准**：管理员打开官网 → 点击想修改的内容 → 原地编辑 → 点发布 → 刷新页面 → 新内容在 30-60 秒内可见
- **关键约束**：
  - 已有 Next.js + Supabase + Vercel 技术栈
  - 编辑体验偏好：所见即所得（直接在页面上点击编辑）
  - 编辑范围：全站内容（项目卡片、文案、日志、团队信息等）
- **隐藏假设**（已确认）：
  1. 双语内容（中英文）需要在同一编辑界面中维护
  2. 当前数据写死在 i18n JSON 文件中，需迁移到 CMS 内容集合
  3. 所见即所得编辑体验通过 Tina CMS 实现，非自建

---

## 发散思考

讨论过程中考虑了以下方向：

### 角度 1: 可视化 Git CMS（Tina CMS / Keystatic）✅ 选定

- **是什么**：Tina CMS 或 Keystatic 等工具——专门做"在页面上直接点击编辑"。内容存在 git 仓库的 Markdown/JSON 文件里，编辑后自动 git 提交并触发 Vercel 部署。
- **为什么相关**：完美匹配"所见即所得"体验需求。Tina CMS 的核心卖点就是"在你的真实页面上编辑"。不需要额外数据库。
- **权衡**：
  - ✅ 开箱即用的所见即所得编辑体验
  - ✅ 不需要额外数据库，利用现有 Vercel 部署流程
  - ✅ 免费开源
  - ⚠️ 编辑后仍需 Vercel 部署（30-60 秒）——用户确认可接受

### 角度 2: Supabase + 自建管理后台 ❌ 排除

- **是什么**：在 Next.js 中新增 `/admin` 后台，用 Supabase 存储所有内容，前台从数据库读取。
- **排除原因**：开发工作量大（数据迁移 + 后台 UI + 所见即所得编辑），而 Tina CMS 能提供更成熟的编辑体验且无需自建。

### 角度 3: 折中方案（Supabase + 表单后台 + 仅关键内容动态化）⏸️ 降级备选

- **是什么**：只把频繁变动的内容（项目卡片、日志）做成可编辑，其余继续写在代码里。后台采用表单式编辑。
- **保留原因**：如果未来发现某些内容确实需要秒级生效（如限时活动），可单独把该类内容迁移到 Supabase，与 Tina CMS 共存。

### 角度 4: 第三方 Headless CMS（Strapi / Sanity）❌ 排除

- **是什么**：Strapi（自部署）或 Sanity（云服务）等专业内容管理平台，通过 API 给 Next.js 喂数据。
- **排除原因**：引入额外服务和成本，管理后台不是所见即所得体验，对 2-3 人团队过度设计。

## 最终方案

### 做什么

1. **引入 Tina CMS** 作为官网的内容管理系统
2. **全站内容可编辑**：项目卡片、首页文案、愿景描述、观测日志、团队信息、联系方式等
3. **所见即所得编辑体验**：管理员在真实的官网页面上直接点击内容进行编辑
4. **双语内容管理**：中英文内容在同一个编辑界面中维护
5. **自动部署**：编辑发布后自动 git 提交 → Vercel 部署 → 30-60 秒后官网更新
6. **管理员认证**：用户和合伙人通过账号登录后台（Tina CMS 自带认证机制）

### 怎么做

- **Tina CMS** 集成到现有 Next.js 项目中
- 当前写在 `src/i18n/locales/zh-CN.json` 和 `en-US.json` 中的内容迁移为 Tina CMS 的内容集合（content collections）
- 管理员访问 `/admin` 进入编辑模式，页面上的文案变成可点击编辑状态
- 内容存储在 git 仓库的 Markdown/JSON 文件中
- Supabase 继续用于用户认证等现有功能，CMS 内容不走数据库

### 不做什么（明确排除）

- ❌ 不搭建 Supabase 数据库存储 CMS 内容（除非未来有秒级生效需求）
- ❌ 不做独立的后台管理界面（所有编辑在页面上完成）
- ❌ 不做复杂的权限分级（两人够用就行）
- ❌ 不引入第三方 Headless CMS（Strapi / Sanity 等）

### 后续步骤

1. 调研 Tina CMS 与 Next.js + next-intl 的集成方案细节
2. 设计内容集合（collections）的数据结构，覆盖全站内容
3. 先在项目卡片模块做原型验证（MVP）
4. 验证通过后逐步迁移其余内容模块

## 决策记录

| 决策 | 原因 | 被排除的替代方案 |
|------|------|-----------------|
| 选用 Tina CMS | 完美匹配所见即所得体验，免费开源，利用现有 Vercel 部署 | Strapi/Sanity（过重）、自建后台（开发量大） |
| 全站内容可编辑 | 用户需求明确，覆盖项目卡片、文案、日志、团队等所有内容 | 仅部分动态内容可编辑（折中方案降级备选） |
| 内容存储在 git 文件中 | Tina CMS 原生方式，无需额外数据库 | Supabase 数据库存储（增加复杂度） |
| 接受 30-60 秒部署延迟 | 用户确认可接受，Vercel 部署足够快 | 实时生效需要 Supabase（当前不需要） |
| Supabase 保留作为降级备选 | 未来如需秒级生效可迁移部分内容 | 现在就引入双系统（过度设计） |

## 遗留问题

1. **Tina CMS + next-intl 兼容性**：Tina CMS 对多语言的支持方式需要与 next-intl 的 i18n 结构对齐，可能需要调整数据结构
2. **认证方式**：Tina CMS 的认证机制（GitHub OAuth / Tina Cloud）需要确认哪种适合当前团队
3. **现有 Aetherwit Town 子页面**：`/projects/aetherwit-town` 的长内容页面是否也纳入 CMS 管理
4. **图片/媒体管理**：项目卡片如果需要配图，媒体文件的上传和管理方式待定
